🏗️ Technical Architecture

GPT-4 is a large-scale Transformer with a decoder-only architecture, trained with a combination of supervised fine-tuning and RLHF. OpenAI has not published the parameter count for GPT-4, but informed estimates range from 1 to 1.8 trillion parameters in a Mixture-of-Experts (MoE) configuration — meaning the full model is large but only a subset of parameters is active for any given token, keeping inference costs manageable.

GPT-4o ("o" for omni) represents a step toward native multimodality — unlike earlier GPT-4V which used separate vision modules, GPT-4o is trained end-to-end on text, images, and audio together. This enables richer cross-modal reasoning and the natural voice conversation mode. The architecture handles audio as compressed audio tokens rather than first transcribing to text.

📜 Historical Context

GPT-1 (2018): Demonstrated that pre-training on large text corpora produces transferable representations. 117M parameters. Relatively modest but conceptually foundational.

GPT-2 (2019): 1.5B parameters. OpenAI controversially refused to release the full model initially, citing misuse concerns. In retrospect, this was more about attention than genuine risk — GPT-2 is trivially outclassed by modern models.

GPT-3 (2020): 175B parameters. The breakthrough moment. Few-shot learning abilities surprised even the researchers. API access launched the LLM application era.

ChatGPT (Nov 2022): GPT-3.5 fine-tuned with RLHF and released with a consumer chat interface. 100 million users in two months. The product that brought AI to the mainstream and triggered the current AI boom.

GPT-4 (Mar 2023): Multimodal. Passed the bar exam in the top 10th percentile. Outperformed all previous models on standard benchmarks by a wide margin. Catalysed unprecedented competitive response from Google, Anthropic, and Meta.

💰 Pricing note: OpenAI offers significant discounts through Batch API (50% off) and through cached prompt pricing. Real cost for most applications is lower than the base rate.

🏗️ Constitutional AI — Anthropic's Unique Approach

Claude is trained using Constitutional AI (CAI), Anthropic's alignment methodology. Rather than relying exclusively on human preference ratings (RLHF), CAI uses a written "constitution" — a set of principles ("be helpful, be harmless, be honest") — to guide the model to critique and revise its own outputs.

This produces several unique properties: Claude is more likely to engage thoughtfully with edge cases rather than refuse reflexively. It expresses uncertainty rather than confabulating. It can disagree with users when warranted. These behaviours are deliberate design choices, not accidents — Anthropic argues that a truly helpful AI must be willing to say "I don't know" and "I disagree" when appropriate.

Anthropic was founded by Dario Amodei, Daniela Amodei, and others who left OpenAI in 2021 over differences in approach to AI safety. The company has published significant research on interpretability (understanding what's happening inside neural networks) as a route to safer AI.

💡 What Claude Does Differently

Nuance and honesty: Claude is noticeably more willing to express opinions, acknowledge limitations, and push back on false premises than competitors. Where other models might agree with a factually incorrect premise to be "helpful," Claude will correct it.

Long-document performance: The 200K context window combined with strong attention to detail across long contexts makes Claude particularly effective for tasks like "read this entire 500-page document and answer questions about it" — it tends to actually use information from throughout the document, not just the beginning and end.

Coding excellence: Claude 3.5 Sonnet leads the SWE-bench coding benchmark (resolving real GitHub issues) as of late 2024. Combined with Claude Code (agentic coding tool), it represents the state-of-the-art in AI-assisted software development.

🏗️ Why 1M Tokens Changes Everything

A 1 million token context is not just "bigger" — it's a qualitative shift. You can feed Gemini the entire codebase of a mid-sized application (200,000 lines of code = ~700K tokens) and ask architectural questions about it. You can provide an entire year of customer service transcripts. You can input a feature-length film (as frames) plus its screenplay and ask the model to fact-check the subtitles against the video.

The challenge: attention over 1M tokens is computationally expensive (attention scales O(n²) with context). Google uses "ring attention" and other infrastructure techniques to make this tractable. The quality of information retrieval at the extremes of the context window (especially the middle) is still an open challenge — the "lost in the middle" problem is more acute at 1M tokens than at 200K.

🔬 Architecture: The Mixture-of-Experts Design

Gemini 1.5 reportedly uses a Mixture-of-Experts (MoE) architecture — the same approach speculated for GPT-4. Rather than one dense model, MoE uses many specialised "expert" sub-networks, with a learned routing mechanism selecting which experts to activate for each token. This means the "full" model can be enormous while per-token compute stays manageable.

Google has the unique advantage of training on Google-exclusive data — YouTube video transcripts, Google Books, Google Scholar papers, Gmail (anonymised), and Google Search — in addition to public web data. This proprietary data advantage is significant, particularly for tasks involving web knowledge, multilingual content, and multimedia understanding.

🔓 Why Open Source Matters

When Meta releases Llama weights, any researcher, startup, or individual can download and run a frontier-capable model with no API fees, no rate limits, and complete privacy. This has several important implications. For privacy-sensitive applications (healthcare, legal, personal data), running a local model means data never leaves your infrastructure. For cost-sensitive applications (high-volume inference), avoiding API fees can save millions annually at scale.

The open ecosystem also means the community can fine-tune Llama for specialised tasks — there are thousands of Llama derivatives on HuggingFace, fine-tuned for medical note-taking, legal document analysis, specific coding languages, and regional languages. This specialisation is impossible with closed models.

🚀 Running Llama locally: Install Ollama, then run ollama run llama3.1:8b to get a capable LLM running entirely on your machine. No account, no API key, no data leaving your device.

🏗️ The MoE Innovation — Mixtral

Mixtral 8x7B was a landmark moment in open AI: a Mixture-of-Experts model where 8 expert networks each have 7B parameters (56B total), but only 2 experts are activated per token (14B effective per forward pass). This means you get GPT-3.5 level performance with the computational cost of a 14B model at inference — a remarkable efficiency gain.

The routing mechanism — deciding which 2 of 8 experts handles each token — is itself a small learned network. Different experts specialise in different types of content without explicit supervision: some handle code, others multilingual text, others creative writing. This emergent specialisation from a purely learned router is one of the most interesting phenomena in modern ML.

🧠 How Reasoning Models Work

Standard LLMs (GPT-4, Claude, Gemini) generate responses in a single forward-pass chain, producing tokens one at a time with no explicit planning. Reasoning models like o1 use a different approach: before generating the final response, the model generates a long internal "chain of thought" — working through the problem step by step, trying different approaches, catching its own errors, and backtracking when needed.

This internal reasoning happens in "reasoning tokens" — tokens that are generated but not shown to the user (the model "thinks" before answering). The final response is generated only after this internal deliberation. For hard problems (competitive maths, logical puzzles, complex code), this extended reasoning can dramatically improve accuracy — o1 solved 83% of 2024 International Mathematical Olympiad problems, versus 13% for GPT-4o.

The cost is time: generating thousands of internal reasoning tokens before responding means latency of 10–60+ seconds for complex problems. This makes o1 inappropriate for conversational applications (too slow) but powerful for analytical tasks where accuracy matters more than speed.

📊 When to use o1: Only when accuracy on hard problems truly matters — competitive programming, formal proof verification, complex analytical tasks. For everything else, GPT-4o or Claude 3.5 Sonnet are faster, cheaper, and equally capable.

Performance

🏆 Benchmark Performance

How AI models score on standardised tests — and what those tests actually measure.

⚠️ Benchmarks measure narrow aspects of AI capability. A model that tops a benchmark isn't necessarily better for your specific use case. Use benchmarks as orientation, not gospel. Also note: labs sometimes train on benchmark data, inflating scores.

MMLU — General Knowledge (% correct)

Massive Multitask Language Understanding. 57 subjects including STEM, humanities, social science. 4-choice multiple choice. Tests broad factual knowledge. Human expert baseline: ~89%.

HumanEval — Coding (% pass)

164 Python programming problems. Measures ability to write functional code from docstrings. Tests algorithmic thinking and Python knowledge.

MATH — Mathematical Reasoning (%)

12,500 competition mathematics problems across 7 difficulty levels. Tests step-by-step mathematical reasoning. This is where reasoning models (o1) have the largest advantage. Human expert baseline: ~90%.

SWE-bench Verified — Real-World Coding

% of real GitHub issues resolved by the AI autonomously. The most practical coding benchmark — uses actual open-source bugs, not artificial problems. Measures real-world software engineering capability.

Understanding Benchmark Limitations

Benchmark contamination: If training data contains benchmark questions (even indirectly), scores are inflated. This is a real concern for all major benchmarks. True "unseen" evaluation is difficult to achieve at scale.

Single-benchmark thinking is misleading: A model that tops MMLU might underperform on creative writing, code review, or nuanced reasoning. Benchmarks measure specific skills. Always evaluate on your actual use case.

The "vibes" gap: Practitioners frequently report that their subjective preference for a model doesn't match benchmark rankings. User experience, consistency, following complex instructions precisely, and appropriate tone matter enormously in practice — and are harder to benchmark.
