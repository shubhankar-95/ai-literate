What Does "Integrating AI" Mean?
The difference between using AI and embedding AI into your workflow.
💡
The Key Insight

A generic AI model like Claude knows a lot about the world — but it knows nothing about your world. It doesn't know your company's data, your codebase, your customers, or your internal processes.

AI integration is the practice of giving AI access to YOUR context so it can do useful work with YOUR data, in YOUR systems, following YOUR rules.

❌
Without integration

"Write a summary of our Q3 performance."

→ Claude makes up plausible-sounding numbers. Useless.

✅
With integration

"Write a summary of our Q3 performance."

→ Claude fetches real data from your BI tool via MCP, reads the actual numbers, writes an accurate summary. Valuable.

🎯

The hierarchy: Generic AI → AI with your documents → AI with your data → AI with your tools → AI that can act on your behalf. Each step multiplies the value you get.

🔌 Protocol

Model Context Protocol

The open standard for connecting AI to any tool, database, or service. Built by Anthropic, supported everywhere.

What is MCP?
Think of MCP as a universal adapter — like USB-C for AI tools.
🔌
The USB-C Analogy

Before MCP, every AI integration was custom: a one-off connector between Claude and, say, GitHub. This required specialized code every time you wanted to connect a new tool. Time-consuming and brittle.

MCP (Model Context Protocol) is a standard that lets any AI model connect to any tool that speaks the protocol. Claude + MCP = Claude can read your database, check your calendar, post to Slack, or query any API — as long as there's an MCP server for it. And there usually is.

🌐

Open source: MCP is an open protocol (github.com/modelcontextprotocol). Anyone can build an MCP server, and hundreds already exist for every popular tool.

Security: Use ${ENV_VAR} syntax for secrets — never hardcode API keys in mcp.json. Keep this file out of version control if it contains sensitive paths.

Building a Custom MCP Server
Create your own MCP server to connect Claude to any internal API or data source.
🏗️
Custom MCP Server in TypeScript
Expose your internal API to Claude
// mcp-server/src/index.ts import { Server } from "@modelcontextprotocol/sdk/server/index.js"; import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js"; import { CallToolRequestSchema, ListToolsRequestSchema } from "@modelcontextprotocol/sdk/types.js"; const server = new Server( { name: "my-company-mcp", version: "1.0.0" }, { capabilities: { tools: {} } } ); // Define your tools server.setRequestHandler(ListToolsRequestSchema, async () => ({ tools: [ { name: "get_customer_data", description: "Get customer info from CRM by email or ID", inputSchema: { type: "object", properties: { customer_id: { type: "string", description: "Customer ID or email" } }, required: ["customer_id"] } }, { name: "search_knowledge_base", description: "Search internal docs and knowledge base", inputSchema: { type: "object", properties: { query: { type: "string" }, limit: { type: "number", default: 5 } }, required: ["query"] } } ] })); // Handle tool calls server.setRequestHandler(CallToolRequestSchema, async (request) => { if (request.params.name === "get_customer_data") { const { customer_id } = request.params.arguments as { customer_id: string }; const data = await fetchFromCRM(customer_id); // your internal API return { content: [{ type: "text", text: JSON.stringify(data) }] }; } // ... handle other tools }); const transport = new StdioServerTransport(); await server.connect(transport);
🚀

Real power: A custom MCP server connected to your CRM, internal knowledge base, and ticketing system turns Claude into an AI that actually knows your business — not just generic knowledge. This is the difference between a toy and a tool.

🎼 Multi-Agent

AI Orchestration

How to coordinate multiple AI agents to tackle complex, multi-step tasks that no single agent can handle alone.

What is Agent Orchestration?
The analogy: one conductor directing many musicians.

A single AI agent is great for focused tasks. But complex work often requires multiple specialized steps: research, then analysis, then writing, then review. Orchestration means coordinating multiple AI agents, each doing what it does best.

Think of it like a kitchen: there's a head chef (orchestrator) who assigns tasks to specialist chefs (sub-agents) — one handles prep, one handles the stove, one handles plating. The final dish is better than what any single chef could produce.

🎼
Simple example: Research → Write → Review

Orchestration Patterns
Common patterns for coordinating AI agents effectively.

🔄 Sequential Pipeline

PATTERN 1

Agents execute one after another, each taking the previous output as input. Best for: content production, data transformation, document processing workflows.

Input → [Research Agent] → [Analysis Agent] → [Writing Agent] → [QA Agent] → Output
# Claude Code prompt for sequential pipeline "Complete this research report in 4 sequential steps: Step 1: Search for the top 5 recent papers on [topic] and summarize findings. Step 2: Identify the 3 most significant trends from your research. Step 3: Write a 500-word executive summary for C-suite audience. Step 4: Review your own draft: check for accuracy, clarity, missing data. Revise."

⚡ Parallel Fan-Out

PATTERN 2

Orchestrator splits work across multiple agents simultaneously, then aggregates results. Best for: multi-source research, running tests in parallel, analyzing many documents at once.

             ┌→ [Agent: Analyze Competitor A] →┐
Input → Orch ┼→ [Agent: Analyze Competitor B] →┼→ [Aggregator] → Report
             └→ [Agent: Analyze Competitor C] →┘

🔁 Iterative Loop

PATTERN 3

Agent runs, evaluates its own output, and iterates until a quality criterion is met. Best for: code generation (run + fix loop), document quality improvement, data cleaning.

Start → [Generate] → [Evaluate] → Pass? → Done
                        ↑              ↓ Fail
                        └──── [Revise] ←┘

🧪 Specialist Routing

PATTERN 4

Orchestrator classifies the request and routes it to the best specialist agent. Best for: customer support (billing → billing agent, tech support → tech agent), multi-domain Q&A systems.

Input → [Router/Classifier] → billing? → [Billing Agent]
                           → tech?    → [Tech Support Agent]
                           → legal?   → [Legal Compliance Agent]

import anthropic import asyncio client = anthropic.Anthropic() async def run_agent(system: str, prompt: str, model="claude-sonnet-4-5") -> str: """Run a single agent with given system prompt and task.""" msg = client.messages.create( model=model, max_tokens=2000, system=system, messages=[{"role": "user", "content": prompt}] ) return msg.content[0].text async def research_pipeline(topic: str) -> str: # Step 1: Orchestrator plans plan = await run_agent( system="You are a research director. Break research tasks into 3 focused subtasks.", prompt=f"Create a research plan for: {topic}. Output as JSON list of subtasks.", model="claude-opus-4-5" # Use stronger model for orchestration ) # Step 2: Run specialist agents in parallel tasks = [ run_agent("You are a technical research specialist.", f"Research: {subtask}") for subtask in parse_plan(plan) ] results = await asyncio.gather(*tasks) # Parallel execution # Step 3: Synthesis agent aggregates synthesis = await run_agent( system="You are a research synthesizer. Combine findings into a cohesive report.", prompt=f"Synthesize these research findings:\n{chr(10).join(results)}" ) # Step 4: QA agent reviews final = await run_agent( system="You are a QA editor. Model strategy: Use Claude Opus for orchestration (needs best reasoning to plan and coordinate) and Claude Sonnet for sub-agents (faster and cheaper for execution tasks). This keeps cost reasonable while maximizing quality.

🏗️ Design

AI Architecture Patterns

Proven design patterns for building reliable, scalable AI-powered systems — from simple integrations to enterprise deployments.

The Core Patterns
Every production AI system uses one or more of these foundational patterns.
📞
1. Tool Use / Function Calling

AI decides when to call external tools (APIs, databases, functions) to get information or take actions. The model orchestrates the sequence of tool calls to complete a task.

Use when: AI needs real-time data, needs to perform actions, or needs to compute something outside its training knowledge.

💡

MCP servers are the modern way to provide tools to Claude. Each MCP server exposes a set of typed functions Claude can call.

📚
2. RAG (Retrieval-Augmented Generation)

Before answering, the AI retrieves relevant documents from a knowledge base and uses them as context. Solves the problem of the AI not knowing your proprietary data.

Use when: You have a large document corpus (policies, docs, past conversations) and need the AI to answer questions accurately from it.

📌

RAG prevents hallucination by giving the AI ground truth to cite. The key challenge is good retrieval — getting the RIGHT chunks, not just similar ones.

🧠
3. Memory Architecture

AI systems maintain different types of memory: in-context (current conversation), external (vector DB), episodic (past interactions), semantic (facts about the user).

Use when: Building assistants that need to remember user preferences, past decisions, or accumulated knowledge across sessions.

🔒
4. Human-in-the-Loop

AI proposes, human approves. Critical checkpoints where human judgment is required before AI proceeds. Prevents costly mistakes in irreversible actions.

Use when: Actions have real-world consequences (sending emails, making purchases, modifying production data). Always build approval gates.

Architecture Decision Framework
How to choose the right pattern for your use case
Do you need the AI to access YOUR data?
├── No  → System prompt + prompting is enough
└── Yes → How often does it change?
         ├── Rarely (docs, policies) → RAG / vector search
         └── Frequently (live DB, APIs) → Tool use / MCP

Do you need AI to TAKE ACTIONS, not just answer?
├── No  → Pure generation patterns are sufficient
└── Yes → Do actions need human approval?
         ├── Yes → Human-in-the-loop architecture
         └── No  → Autonomous agent (with careful scope limits)

Is the task too complex for one AI call?
├── No  → Single-turn or conversational pattern
└── Yes → Break into steps
         ├── Sequential? → Pipeline pattern
         ├── Parallel?   → Fan-out orchestration
         └── Iterative?  → Loop with evaluator
🔍 Retrieval

RAG & Memory

How to make AI know your data — from basic document retrieval to long-term memory systems.

Why AI Needs Retrieval
The fundamental problem RAG solves.

Claude knows a lot about the world up to its training cutoff, but it knows nothing about:

• Your internal documentation · Your product specifications · Your customer data · Your past project notes · Anything that happened after its training cutoff

RAG (Retrieval-Augmented Generation) solves this: before Claude answers, your system retrieves the most relevant documents from your knowledge base and includes them in the context. Claude then answers based on YOUR data, not just training.

WITHOUT RAG:   User Question → Claude → Answer (from training only)

WITH RAG:      User Question → Search KB → Find relevant docs
                                              ↓
                              Question + Relevant Docs → Claude → Grounded Answer
Building a Basic RAG System
The four components every RAG system needs.
1

Chunk your documents

Split large documents into smaller chunks (typically 256–512 tokens). Each chunk should be semantically coherent — don't split mid-sentence or mid-thought. Overlap chunks slightly to avoid losing context at boundaries.

2

Embed chunks as vectors

Use an embedding model (like OpenAI text-embedding-3-small or Voyage AI) to convert each chunk into a vector — a mathematical representation of its meaning. Similar chunks will have similar vectors.

# Using Anthropic's partner Voyage AI for embeddings import voyageai vo = voyageai.Client() embeddings = vo.embed(chunks, model="voyage-3").embeddings
3

Store in a vector database

Popular options: Pinecone (managed, easy), Weaviate (open source, powerful), Chroma (local dev), pgvector (if you already use Postgres). At small scale, even a JSON file works.

4

Retrieve and generate

At query time: embed the user's question, find the most similar chunks (cosine similarity), include them in Claude's context, and ask Claude to answer using those chunks.

# At query time: query_embedding = embed(user_question) relevant_chunks = vector_db.search(query_embedding, top_k=5) context = "\n\n".join(relevant_chunks) response = claude.messages.create( system="Answer based on the context provided. If unsure, say so.", messages=[{"role": "user", "content": f"Context:\n{context}\n\nQuestion: {user_question}"}] )
Advanced RAG: Beyond Basic Retrieval
Techniques that dramatically improve RAG quality in production.
🎯
Hybrid Search

Combine vector similarity search (semantic) with BM25 keyword search (lexical). Vector search finds conceptually similar content; keyword search finds exact matches. Combining both dramatically improves recall.

# Reciprocal Rank Fusion combines both vector_results = vector_search(query) keyword_results = bm25_search(query) combined = reciprocal_rank_fusion( vector_results, keyword_results )
🔁
Re-ranking

After initial retrieval, use a re-ranker model to re-score the top-N results by how well they actually answer the query. Cohere Rerank or a cross-encoder model identifies the truly relevant chunks.

results = retrieve(query, top_k=20) # Re-rank narrows to the best 5 reranked = cohere.rerank(query, results, top_n=5)
🧩
Query Decomposition

For complex questions, have Claude break the query into sub-questions, retrieve for each, then synthesize. A question like "Compare our Q3 to Q2 and explain the difference" needs multiple retrieval passes.

💾
Contextual Memory

For long-running assistants: store conversation summaries, user preferences, and past decisions in a memory store. Each new conversation retrieves relevant memories to maintain context without sending the full history.

🛠️ Hands-On

Build Your Own AI System

Step-by-step guides for building real AI-powered applications from scratch.

Project 1: Company Knowledge Bot
A Slack bot that answers questions from your internal docs — built in a weekend.
1

Collect your documents

Gather: company handbook, product docs, past meeting notes, onboarding materials, FAQs. Export to plain text or Markdown. Aim for 100–500 documents to start.

2

Build the RAG pipeline

Use LlamaIndex or LangChain to handle chunking + embedding + storage. Connect to Chroma (local) or Pinecone (cloud). Load your documents with 5 lines of code using pre-built loaders.

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader docs = SimpleDirectoryReader("./company-docs").load_data() index = VectorStoreIndex.from_documents(docs) query_engine = index.as_query_engine(llm=Anthropic(model="claude-sonnet-4-5"))
3

Add a Slack interface

Use Slack's Bolt framework. When a user @mentions the bot, send their message to the query engine, stream the response back. Add to any channel in minutes.

4

Deploy and monitor

Deploy on Railway, Render, or AWS Lambda. Log every query and response. Review weekly — bad answers reveal gaps in your docs. Add missing content to the knowledge base.

Project 2: Automated Research Agent
An agent that researches topics, synthesizes information, and produces reports — fully automated.
1

Define the research brief template

What sections does your ideal research report have? Define this as your system prompt. "You are a research analyst. Always structure reports as: Executive Summary → Key Findings → Evidence → Recommendations → Sources."

2

Give Claude web research tools

Connect the Brave Search MCP server and Web Fetch MCP server. Claude can now search the web and read pages — actual current information, not just training data.

3

Build the iterative loop

Agent searches → reads sources → identifies gaps → searches more → synthesizes → self-reviews → outputs final report. The loop terminates when Claude decides it has enough evidence.

4

Schedule and distribute

Trigger the agent on a schedule (weekly competitive analysis, daily news digest). Auto-post results to Slack, email, or a shared Google Doc via MCP connectors.

🎯

Start small, prove value, expand. Your first AI integration should solve a painful problem for a small team (3–5 people) and deliver clear time savings. Once proven, scaling to the whole organization becomes much easier — you have evidence, champions, and a working template.
