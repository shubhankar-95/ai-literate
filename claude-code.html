What is Claude Code?
A coding assistant you talk to in your terminal — like having a senior developer always on call.
🤖
Think of it this way
The simplest explanation

Claude Code is an AI you run from the command line (the text-based terminal on your computer). You describe what you want in plain English, and it writes, edits, and runs code for you. It can read your entire project, understand how it all fits together, and make changes across many files at once.

Unlike ChatGPT where you copy-paste code back and forth, Claude Code directly touches your files. It's like the difference between asking for directions and having a navigator sitting in your car.

💬
Old way: ChatGPT

You copy your code → paste into browser → get suggestion → copy back → manually apply → hope it works.

⚡
New way: Claude Code

You type: "Add login with Google to my app" — Claude reads your project, writes the code, and applies it. Done.

💡

You don't need to know how to code to start using Claude Code — but learning the basics helps you give better instructions and review what it produces.

What Claude Code Actually Does
A full-stack agentic coding system — not just autocomplete.
🧠
The Agent Loop
How Claude Code solves problems

Claude Code runs an agent loop: it reads your request, decides what tools to use (read files, write files, run bash, search the web), executes them, observes the results, and continues until the task is done. It's not a one-shot response — it works like a developer iterating on a problem.

Key tools in its arsenal: Read (any file), Write/Edit (surgical edits), Bash (run tests, build, install packages), WebFetch (read docs), Glob/Grep (find code patterns).

✅
What it can do

Refactor an entire codebase · Add features across multiple files · Debug runtime errors by reading logs · Write and run tests · Generate documentation · Set up new projects from scratch · Research APIs via web fetch.

⚠️
What it can't do (yet)

It can't see your screen or browser · Can't interact with GUIs · Won't commit/push without your approval · Can't access services that need your credentials unless you set them up · Doesn't remember previous sessions by default.

Architecture & Internals
How Claude Code is built and why it works so well for engineering tasks.
⚙️
Under the Hood
The technical architecture

Claude Code is built on Anthropic's Claude Sonnet/Opus models, accessed via the Anthropic API. It runs as a Node.js process that manages a tool-calling loop — each iteration sends the conversation + tool results back to the model, which decides its next action.

The model uses a 200K token context window, meaning it can hold the equivalent of ~500 pages of code in working memory simultaneously. For very large codebases, it uses smart file selection (via Glob/Grep) rather than loading everything.

Permissions are controlled at the session level. Claude Code asks for approval before any destructive action (file delete, git reset, package install) unless you've pre-authorized it. The permission model maps directly to Unix file system permissions — it can only touch what your user account can.

# The agent loop conceptually:
# 1. User sends message # 2. Claude picks a tool: Read / Write / Bash / etc. # 3. Tool runs, returns result to Claude # 4. Claude reasons over result, picks next tool # 5. Repeats until task complete # 6. Claude sends final message to user # You can observe this loop in real time with: claude --verbose # shows every tool call live
🔬

SWE-bench insight: Claude 3.5 Sonnet achieved 49% on SWE-bench Verified (a benchmark of real GitHub issues), compared to ~18% for GPT-4o in 2024. This gap is why Claude Code feels materially different for real engineering tasks.

🛠️ Setup

Getting Started

From zero to your first AI-written feature in under 10 minutes.

Step-by-Step Installation
Follow these steps exactly. No prior experience needed.
1

Install Node.js (if you don't have it)

Claude Code requires Node.js version 18 or newer. Go to nodejs.org and download the LTS version. Run the installer and follow the prompts. To verify it worked, open your terminal and type:

node --version # Should show v18.x.x or higher
2

Install Claude Code via npm

npm (Node Package Manager) comes with Node.js. Open your terminal and run:

npm install -g @anthropic-ai/claude-code

The -g flag installs it globally so you can use it from any folder.

3

Get your Anthropic API Key

Go to console.anthropic.com → sign up or log in → API Keys → Create new key. Copy it. This is your "password" that lets Claude Code talk to Anthropic's servers. Keep it private.

⚠️

Never share your API key publicly — in GitHub repos, chat, or anywhere. If leaked, someone else can use your account's credits.

4

Set your API key

Run this command, replacing the placeholder with your actual key:

export ANTHROPIC_API_KEY=sk-ant-your-key-here # To make it permanent (so you don't need to do this every time), # add the above line to your ~/.bashrc or ~/.zshrc file
5

Navigate to your project and start

Go to the folder containing your code, then start Claude Code:

cd ~/my-project # go to your project folder claude # start Claude Code

You'll see a prompt. Type your first request in plain English!

Prompting Tips — What Works & What Doesn't
Every example below shows the bad version, what goes wrong, and the fix.
❌ Mistake 1 — Too vague, no context
❌ What you typed
Fix my code
What goes wrong: Claude doesn't know which file, which function, what the error is, or what "fixed" looks like. It will either ask clarifying questions or guess — and guess wrong. It might refactor things that weren't broken.
✅ What to type instead
In src/api/users.js, the POST /api/users endpoint returns 500 when the email field is missing from the request body. It should return 400 with JSON: {"error": "email is required"}. Fix only this endpoint — don't touch the other routes.
Why it works: File path, endpoint, exact error, expected behaviour, scope boundary. Claude has everything it needs and nothing it doesn't.
❌ Mistake 2 — No scope, Claude touches everything
❌ What you typed
Add dark mode to the app
What goes wrong: Claude will pick a random implementation approach and touch files across the whole project. If your app has 40 components, you'll get 40 edits you didn't review and didn't want.
✅ What to type instead
Add a dark mode toggle to the app. CSS uses custom properties in src/styles/variables.css. Add a data-theme="dark" attribute on <html> when toggled, and add a dark palette block to variables.css. The toggle button goes in src/components/Header.jsx only. Don't touch any other component files yet.
Why it works: You defined the architecture (CSS vars), the pattern (data-theme), the exact files, and what to leave alone. No surprise edits.
❌ Mistake 3 — Pasting an error with no surrounding code
❌ What you typed
TypeError: Cannot read properties of undefined (reading 'map') Why is this happening?
What goes wrong: This error occurs in thousands of different scenarios. Claude gives generic debugging advice ("maybe the data isn't loaded yet") that doesn't apply to your specific code.
✅ What to type instead
Getting this in src/pages/Dashboard.jsx line 42: TypeError: Cannot read properties of undefined (reading 'map') The line is: {userData.orders.map(order => ...)} userData comes from the useUser() hook. Error only happens on first load — works after a refresh. Root cause and fix?
Why it works: File, line, the actual code, the data source, the reproduction pattern. Claude can now diagnose the real problem (async data not yet loaded) and give a precise fix.
❌ Mistake 4 — "Make it better" with no definition of better
❌ What you typed
Can you improve this function?
What goes wrong: "Better" means nothing. Claude might add comments (you wanted speed), rename variables (you wanted error handling), or restructure (you wanted it left alone). You get changes you didn't want.
✅ What to type instead
This function processes 50,000 records and takes ~8s. Optimise for speed only — I don't care about readability. Don't change the function signature or return type. The bottleneck is likely the nested loop on lines 14–22. [paste function]
Why it works: You defined "better" (speed), gave the constraint (keep the signature), pointed at the bottleneck, and said what not to change.
❌ Mistake 5 — One giant task with no checkpoints
❌ What you typed
Build a full user auth system — login, signup, password reset, email verification, OAuth with Google, and a dashboard showing login history.
What goes wrong: Claude writes thousands of lines across dozens of files. If it misunderstands your database schema at step 2, everything after is built on a broken foundation — and you don't find out until the end.
✅ What to type instead
Step 1 only: Set up the database schema for users. Tables: users (id, email, password_hash, created_at, verified_at) and sessions (id, user_id, token, expires_at). Use our existing Prisma setup in prisma/schema.prisma. Don't build any API routes yet — I'll review the schema before we continue.
Why it works: One step, one file, a clear stop point. You review before Claude builds anything on top of it. Each chunk is correctable without unravelling everything after it.
💡
The one habit that prevents most of the above

Before any task touching more than 2 files, add this to the start of your prompt:

"Before making any changes: tell me which files you'll edit, what you'll change in each, and any assumptions you're making. Wait for my approval before you start."

This one sentence catches 80% of misunderstandings before a single line of code is written.

Prompt Engineering for Code
The anatomy of a great prompt — and six anti-patterns that waste tokens and time.
🎯
The CSTE Framework — the anatomy of a complete prompt
Context · Scope · Task · Expectation
C — Context
Stack, file paths, relevant architecture: "Node.js / Express API, Postgres via Prisma, file: src/routes/auth.ts"
S — Scope
What Claude can and cannot touch: "Only edit this file. Don't change the Prisma schema or any test files."
T — Task
What exactly needs to be done: "Add rate limiting — 5 req/min per IP on all /auth routes"
E — Expectation
The success criteria: "Return 429 + JSON {error, retryAfterSeconds}. Run tests after."
# Full CSTE prompt in practice: We have a Node.js/Express API. Auth routes are in src/routes/auth.ts, [C] using express-rate-limit (already installed). Only edit auth.ts — don't touch server.ts or any test files. [S] Add rate limiting: 5 requests per minute per IP on every /auth route. [T] On limit exceeded: return 429 with JSON {error:"Too many requests", [E] retryAfterSeconds: N}. Run npm test after the change and show output.
Six Anti-Patterns — With Full Before/After
These are the patterns that cause Claude to produce useless output, refactor the wrong thing, or quietly go off-track.
Anti-pattern 1 — Describing the solution, not the problem
❌ What you typed
Add a debounce function to the search input so it waits 300ms before calling the API
What goes wrong: You've already decided on the solution (debounce, 300ms). Claude will implement exactly that — even if the real problem is that you're calling the API on every keystroke and a simple controlled input with onBlur would be cleaner. Claude can't push back on a solution it doesn't know you chose arbitrarily.
✅ What to type instead
In src/components/SearchBar.jsx, the API call fires on every single keystroke. This is hammering our /api/search endpoint. What's the right fix? I was thinking debounce at 300ms but open to other approaches. Show me the code once we agree on the approach.
Why it works: You described the problem (API called on every keystroke), your tentative solution, and asked for agreement before implementation. Claude can now suggest debounce, throttle, or a controlled input pattern — and you pick the right one.
Anti-pattern 2 — No constraint on what NOT to change
❌ What you typed
Refactor the UserService class to use async/await instead of .then() chains
What goes wrong: Claude refactors every method in the class — including ones that are passing tests, touch the database layer, and have subtle error-handling logic. You wanted to modernise the syntax; Claude rewrote business logic you didn't touch in two years.
✅ What to type instead
In src/services/UserService.js, refactor only the getUserById() and updateEmail() methods from .then() chains to async/await. Don't touch any other methods, don't change error handling behaviour, don't change the return types. The tests in tests/UserService.test.js should pass without modification after your change.
Why it works: Two specific methods. Three explicit don't-touch rules. A pass/fail test criterion. Claude can't over-reach because you've drawn the exact boundary.
Anti-pattern 3 — Asking for a review without telling Claude what to look for
❌ What you typed
Review this code and tell me if it looks good [pastes 200 lines]
What goes wrong: "Looks good" is unmeasurable. Claude produces a generic checklist — comments about naming conventions, suggests adding a README, notes that tests would be nice. None of which is what you needed before shipping.
✅ What to type instead
Review this payment processing code before we ship it tomorrow. I specifically need you to check: 1. Any place where an exception could be swallowed silently 2. Whether the Stripe webhook signature is being verified correctly 3. Any race condition risk between the charge and the DB write Ignore code style — we have a linter for that. [pastes code]
Why it works: Three specific, high-stakes concerns. One explicit exclusion. Claude reviews what actually matters for production safety, not what's easy to comment on.
Anti-pattern 4 — Using Claude as a search engine for docs
❌ What you typed
How do I use React Query?
What goes wrong: You get a generic tutorial that explains React Query from scratch. It won't match your version, your data-fetching pattern, your error handling, or your file structure. You spend 20 minutes reading before writing one line of code.
✅ What to type instead
We're using React Query v5 with a REST API. I need to add a query for fetching a user's orders from GET /api/orders?userId=X. Our existing query setup is in src/hooks/useUser.ts — match that pattern exactly. Show me just the new hook, nothing else.
Why it works: Specific version, specific endpoint, a reference file to match. Claude produces one hook in your exact style. You copy-paste and move on.
Anti-pattern 5 — Letting Claude write tests without defining what to test
❌ What you typed
Write tests for the checkout function
What goes wrong: Claude writes happy-path tests — the ones that always pass. It won't know about the edge cases that have actually broken in production: the empty cart, the expired coupon code, the item that went out of stock mid-checkout.
✅ What to type instead
Write Jest tests for src/lib/checkout.ts. I need coverage for these cases: - Cart is empty (should throw CartEmptyError) - Coupon code is expired (should return error, not throw) - Item goes out of stock between cart and checkout (should fail gracefully) - Payment succeeds but DB write fails (should refund and return error) - Happy path: valid cart, valid coupon, payment succeeds Use our existing test setup from tests/setup.ts. Mock Stripe using the pattern in tests/payment.test.ts.
Why it works: You defined the exact cases — especially the failure modes you already know about. The tests Claude writes will actually catch real bugs, not just verify that working code works.
Anti-pattern 6 — Not giving Claude a way to know when it's done
❌ What you typed
Add caching to the dashboard API calls
What goes wrong: Claude adds Redis caching, installs a package, modifies 4 files, and sets a 5-minute TTL. You wanted a simple in-memory cache with a 30-second TTL. There's no way for Claude to know the task is done — or that it went in a completely different direction than you intended.
✅ What to type instead
Add in-memory caching to GET /api/dashboard in src/routes/dashboard.ts. Use a plain JS Map — no new packages. Cache key: the userId. TTL: 30 seconds. If cache hit, return cached response and log "cache hit: [userId]". After making the change, test it by calling the endpoint twice in quick succession and show me the logs to confirm caching is working.
Why it works: Implementation specified (Map, not Redis), no new packages, cache key, TTL, log message, and a concrete verification step. Claude knows exactly when it's done and can prove it.

Advanced Prompting — Large Codebases & Production Systems
The patterns that matter when the codebase is complex, the stakes are high, and one wrong prompt costs hours.
Anti-pattern 7 — Asking for a change without explaining the architecture
❌ What you typed
Add logging to the order service
What goes wrong: Claude picks a logging library (probably winston or console.log), adds it to one file, and invents a log format. Your codebase already has a centralised logger, a log correlation ID pattern, and a team convention for log levels. Claude doesn't know any of that — and you won't catch the mismatch until code review.
✅ What to type instead
Add structured logging to src/services/OrderService.ts. We use our internal logger: import { logger } from 'src/lib/logger'. Log pattern to follow: logger.info('order.created', { orderId, userId, total }). Log these events: order created, payment charged, fulfilment queued, any error (log level 'error' with the full error object). Don't add any new logging package — only use our existing logger.
Why it works: You gave the import path, the exact call signature, the event names to log, the log levels, and explicitly prevented new package installs. Claude slots into your existing system rather than inventing a new one.
Anti-pattern 8 — Database migrations without guardrails
❌ What you typed
Add a status column to the orders table
What goes wrong: Claude writes a migration that adds the column but forgets a default value — your 2 million existing rows now have NULL in a non-nullable column. Or it uses the wrong migration tool. Or runs the migration instead of just generating it. On a production database, any of these is a serious incident.
✅ What to type instead
Generate a Prisma migration to add a status column to the orders table. Type: enum OrderStatus {PENDING, PROCESSING, SHIPPED, CANCELLED}. Default value for existing rows: PENDING. The migration should be backwards-compatible — existing rows must not break. Only generate the migration file in prisma/migrations/ — do NOT run it. I will review and run it manually. After generating, also update the Order model in prisma/schema.prisma.
Why it works: Tool specified (Prisma), enum values defined, default for existing rows, backwards-compat requirement, and an explicit "generate but do not run" instruction. You stay in control of when the migration touches the database.
Anti-pattern 9 — "Make it faster" on a slow query
❌ What you typed
This query is slow, optimise it [pastes SQL]
What goes wrong: Claude rewrites the query using techniques that look good syntactically but may not help — or actively hurt — without knowing your indexes, table sizes, query plan, or Postgres version. Generic SQL optimisation advice is often wrong for your specific schema.
✅ What to type instead
This query takes ~4s on our orders table (8M rows, Postgres 15). Indexes that exist: orders(created_at), orders(user_id), orders(status). EXPLAIN ANALYZE output: [paste EXPLAIN output] Goal: under 200ms. Suggest changes — indexes, query rewrite, or both. Don't change the result set. Explain the trade-offs before I implement anything.
Why it works: Table size, Postgres version, existing indexes, EXPLAIN output, a target latency, and a constraint on the result set. Claude now has what a DBA would need to give real advice — not guesses.

# In your CLAUDE.md — hard limits that Claude will see on every session ## BOUNDARIES — Non-negotiable ### Files you may NEVER touch: - .env, .env.*, any secrets file - prisma/migrations/ — generate only, never run - infrastructure/ (Terraform, k8s configs) - Any file not under src/ or tests/ ### Actions you may NEVER take: - Run database migrations - Deploy or push to any remote - Delete files (ask me first) - Install packages without telling me first ### When uncertain: Stop and ask.
